# 站点预设作战手册

适用于 `scripts/init_site_sign.py --preset ...` 生成脚本后的落地阶段。

目标不是解释“预设名字是什么意思”，而是告诉你：

- 这个预设下，第一版脚本该优先做什么
- 哪些事情不要一开始就做错
- 最小闭环如何成立
- 哪些验证信号最可信

---

## 1. `generic`

### 适用站点

- 还没完全看清站点模式
- 只有少量线索，无法先确定 `httpx` 还是 `nodriver`

### 第一优先级

1. 先抓真实登录/签到请求
2. 先确认权威成功判据
3. 再决定是否需要浏览器态

### 最小闭环

1. 登录态检查
2. 执行动作
3. 权威状态复核

### 常见误区

- 一开始就选 `nodriver`，结果其实 `httpx` 足够
- 一开始就选 `httpx`，结果接口需要浏览器态 token
- 还没确认成功判据，就先写一堆登录逻辑

### 优先验证

1. 接口能否直连
2. 浏览器内能否更稳定返回 200
3. 成功状态是否可二次复核

---

## 2. `browser-risk`

### 适用站点

- 登录前有 WAF / Probe / TencentCaptcha
- 登录后还有极验 / 点选 / 短信 / 二次风控
- 最终业务走浏览器态或 App API

### 第一优先级

1. 拆阶段，不要把所有验证码混在一起
2. 先实现 **WAF 放行判据**
3. 再实现 **登录风险验证**
4. 最后实现 **业务 API 闭环**

### 最小闭环

1. WAF 通过
2. 登录页 DOM 就绪
3. 登录验证码成功回填
4. AJAX / 表单登录成功
5. 浏览器态复核成功
6. API 复核成功
7. 签到动作成功
8. 签到后权威复核成功

### 常见误区

- 只看 URL 没变就认为 WAF 已过
- WAF 没放行就开始填表单或发登录 AJAX
- 登录成功后不做 API 复核
- 把 `error_113 / forbidden` 当成普通 offset 问题无限盲调
- 任务列表为空时误报“任务失败”

### 必须落盘

- WAF 截图
- 登录页就绪截图
- 验证码 canvas / overlay / trace
- 成功快照
- 结构化 JSONL 日志

### 优先验证

1. WAF 后是否真的进入真实登录页
2. 验证码成功后是否真回填参数
3. 登录接口是否真返回成功
4. App API 是否真拿到用户信息

---

## 3. `turnstile-json-api`

### 适用站点

- Cloudflare Turnstile
- 最终调用 JSON API
- 接口在浏览器内能成功，脚本直连容易 403 或缺 token

### 第一优先级

1. 先定位验证配置接口 / siteKey 来源
2. 先验证浏览器内 `fetch` 能否成功
3. 再把 token 回填到真实业务请求

### 最小闭环

1. 渲染 Turnstile
2. 获取 token
3. 浏览器内 fetch 登录或签到接口
4. 再调权威状态接口复核

### 常见误区

- 直接拿 HAR 生抄请求，但遗漏浏览器态 token
- 把 Turnstile 验证成功当成业务成功
- 只看 `200`，不看业务 JSON 字段

### 必须落盘

- siteKey 来源
- fetch 请求体与返回体
- token 获取成功样本
- 动作前后权威状态

### 优先验证

1. token 是否真实可用
2. 浏览器态 fetch 与脚本直连行为是否不同
3. JSON 业务字段是否明确成功

---

## 4. `cookie-cloudflare`

### 适用站点

- 主要靠 Cookie 注入
- 但首页、状态页或部分接口会命中 Cloudflare / 简单放行页

### 第一优先级

1. 先验证 Cookie 注入是否真建立登录态
2. 再处理 Cloudflare/放行页
3. 最后再考虑签到动作

### 最小闭环

1. 注入 Cookie
2. 打开权威页
3. 判断是否已登录
4. 执行签到
5. 再读权威状态

### 常见误区

- Cookie 一注入就认为成功
- 登录态没复核就开始签到
- 浏览器态能过但硬要退回 `httpx`

### 必须落盘

- 首页/权威页 HTML 摘要
- 注入 Cookie 后的状态页反馈
- Cloudflare 卡住时的截图或文本

### 优先验证

1. Cookie 是否失效
2. Cloudflare 是否只影响页面，不影响 API
3. 最终权威页是否能读到已登录/已签到状态

---

## 5. `simple-httpx`

### 适用站点

- 无强浏览器依赖
- 接口稳定
- token 可直接提取或根本不需要

### 第一优先级

1. 保持脚本尽量简单
2. 先写权威状态获取
3. 再写签到请求
4. 最后做动作后复核

### 最小闭环

1. 权威页 / 权威接口获取用户状态
2. 提交签到动作
3. 再次获取权威状态

### 常见误区

- 看到 `200` 就算成功
- 从 HTML 文案硬猜积分变化
- 因为偶发页面复杂就过早引入浏览器

### 必须落盘

- 动作前状态
- 签到原始响应
- 动作后状态
- 成功快照

### 优先验证

1. 是否真需要 `nodriver`
2. 权威状态是否能明确区分“已签到”和“新签到成功”
3. 接口是否存在更稳定 JSON 信号

---

## 统一规则

无论哪个 preset，都优先遵守：

1. **登录成功 ≠ 签到成功**
2. **验证码通过 ≠ 业务成功**
3. **HTTP 200 ≠ 最终成功**
4. **必须做动作前/动作后权威复核**
5. **成功样本要固化成快照**
