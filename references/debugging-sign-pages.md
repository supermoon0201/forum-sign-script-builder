# 签到页调试与异常排查

当站点脚本已经基本选型完成，但在实跑时出现“页面乱码”“字段提取失败”“已签到但拿不到金额”“验证码偶发失败”等问题时，优先读本文件。

## 1. 页面乱码 / 标题为空 / 正文像压缩流

### 现象

- `resp.text` 里是乱码，例如 `�$d4��=��...`
- 页面标题提取为空或是“未知”
- 明明拿到 200，但正文不像 HTML
- 表单字段、按钮、状态文案全都匹配不到

### 优先排查

1. 看响应头：
   - `content-encoding`
   - `content-type`
2. 看原始字节前缀：
   - 打印 `resp.content[:32].hex()`
3. 看页面标题与纯文本前 100~300 字符：
   - 如果标题为空、正文不可读，优先怀疑压缩解码问题

### 常见根因

- 运行环境缺少 Brotli 解码依赖
- 服务端返回了 `br`，而客户端没有正确解压
- 某些最小 Python / 青龙环境与本机依赖不一致

### 推荐处理

- 优先把 `Accept-Encoding` 降到：
  - `gzip, deflate`
- 必要时主动禁用 `br`
- 日志里保留：
  - 页面标题
  - `content-encoding`
  - `content-type`
  - 原始字节前缀

### 不要做

- 还没确认编码问题，就直接认定“站点改版”
- 用乱码正文去做状态判断

## 2. 动态登录表单字段提取失败

### 现象

- 用户名、密码、验证码字段名不是固定的
- `once`、`csrf`、隐藏字段偶发为空
- 本机能提取，青龙里提取不到

### 常见站点模式

- `name` 是 64 位哈希
- 属性顺序不固定
- 同一个页面有多个 `input[type=text]`
- 中英文 placeholder 不一致

### 推荐处理

不要只写“固定顺序正则”，优先：

1. 先提取真实 `<input ...>` 标签
2. 解析每个标签属性
3. 再按语义识别字段：
   - `type=password` → 密码字段
   - `name=once` + hidden → once
   - `placeholder` 含“用户名 / email / username” → 用户名字段
   - `placeholder` 含“验证码 / captcha” → 验证码字段

### 兜底策略

- 如果两个 `text` 字段都是哈希名：
  - 非验证码 placeholder 的当用户名
  - 另一个当验证码
- 如果字段仍然缺失：
  - 打印页面标题
  - 打印 `<input>` 数量
  - 打印前若干个真实 input 标签

### 不要做

- 假设字段名永远固定
- 假设 `type`、`name`、`placeholder` 的属性顺序固定
- 只在本机验证，不看青龙环境差异

## 3. 从哪里拿“今日奖励金额”和“当前余额”

### 误区

很多站点的签到页只会告诉你：

- 已签到
- 签到成功
- 明日再来

但**不会给出本次奖励金额**，也不一定显示最新余额。

### 推荐优先级

1. 余额页 / 账单页 / 积分流水页
2. 用户资产页 / 个人中心
3. 签到页文字

### 推荐策略

如果账单页同时包含：

- 当前余额
- 今日签到奖励
- 最近一条每日奖励流水

就优先统一从账单页取，避免跨页口径不一致。

### 典型解析方式

- 顶部区域取：
  - `当前账户余额`
- 表格第一条或今日日期对应行取：
  - 类型：`每日登录奖励`
  - 数额：如 `3.0`
  - 余额：如 `31518.0`
  - 描述：如 `20260620 的每日登录奖励 3 铜币`

### 推荐输出

- 今日奖励格式化为：
  - `3 铜币`
- 当前余额格式化为：
  - `31518`

### 不要做

- 从“已签到”文案里硬猜金额
- 用不带日期的旧流水当作今天奖励
- 从多个页面混搭出互相冲突的数据

## 4. 页面文本提取前先清洗

### 现象

直接去标签后的正文里可能混入：

- 内联脚本
- 广告脚本
- 模板字符串
- 注释内容

会导致：

- 页面反馈极长且不聚焦
- 状态判断被污染
- 把脚本字符串误判成业务文案

### 推荐处理顺序

1. 去掉 `<script>...</script>`
2. 去掉 `<style>...</style>`
3. 去掉 HTML 注释
4. 再去标签
5. 再压缩空白字符

### 适用场景

- 提取页面标题
- 提取状态文案
- 打印失败摘要
- 做“已签到 / 未登录 / 余额”文本匹配

## 5. 图形验证码登录的重试策略

### 现象

- 验证码识别偶发错误
- 第一次失败，第二次换张图就成功
- 账号密码本身没问题，但最终登录态校验没通过

### 推荐默认策略

至少支持有限次重试，例如：

- 2~3 次

每次重试都必须重新执行：

1. 重新获取登录页
2. 重新解析动态字段
3. 重新获取验证码图片
4. 重新识别验证码
5. 重新提交表单
6. 再做登录态校验

### 日志里要区分的阶段

- 字段解析失败
- 验证码识别失败
- 登录提交异常
- 登录提交成功但登录态校验未通过

### 不要做

- 重复提交旧验证码
- 重复使用旧 `once` / `csrf`
- 把所有失败统一记成“登录失败”

---

## 6. 推荐诊断日志模板

至少打印这些内容：

- 账号编号
- 当前步骤：
  - 登录态检查
  - 解析登录页
  - 拉取验证码
  - 提交登录
  - 执行签到
  - 二次校验
- 页面标题
- 关键字段名
- 关键接口路径
- 页面反馈前 100~300 字符
- 响应头诊断：
  - `content-encoding`
  - `content-type`
- 原始字节前缀（必要时）
- 权威校验结果

### 目标

让后续排错时，能快速区分：

- 是环境问题
- 是编码问题
- 是页面结构变了
- 是验证码错了
- 还是业务状态本身就是“今日已签到”
