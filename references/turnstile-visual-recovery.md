# Turnstile 视觉点击恢复流参考

这个参考文件只针对一类站点：

- 页面上存在 **可见 Turnstile 复选框**
- 页面 DOM 中通常只有 `hidden input`、`1x1 iframe`、祖先容器等弱信号
- 需要在真实浏览器里用鼠标点击 challenge
- 点击后可能弹出 `Cloudflare人机验证服务加载失败` 一类错误遮罩
- 真正的成功信号来自 `cf-turnstile-response` 等隐藏字段回填，而不是文案变化

典型代表：

- NodeSeek 登录页这类“账号密码登录 + Cloudflare Turnstile + hidden input 回填 + 视觉点击”链路

## 1. 优先级：真实复选框 > 外层容器 > 文案区

视觉点击型 Turnstile 必须区分三层对象：

1. **祖先容器层**
   - 常见形态：`div` / `fieldset` / 祖先表单块
   - 作用：用于估算 challenge 位于哪一条横向带状区域
   - 风险：容器通常比真实复选框大得多，直接点击很容易落到标题、说明文字或输入框区域
2. **组件条带层**
   - 常见形态：包含 Cloudflare logo、提示文案、隐藏 input 邻域的横条
   - 作用：用于判断 challenge 是否视觉就绪
   - 风险：条带仍然不是点击中心，只能作为候选区域
3. **真实复选框层**
   - 常见形态：左侧一个小方形 checkbox
   - 作用：真正允许点击的对象

经验规则：

- 只有锁定了 **真实复选框小方框**，才允许点击
- 如果只命中了祖先容器或条带，但没有命中真实复选框，默认继续等待，不要 fallback 扫点

## 2. 视觉就绪判定要先于点击

当页面里：

- 已有 `cf-turnstile-response` hidden input
- 已有祖先容器矩形
- 但截图里 challenge 仍是空白占位

此时不要点击。

建议做法：

1. 先截图 challenge 所在左侧区域
2. 计算简单视觉信号，例如：
   - 边缘数量
   - 非纯白像素比例
3. 只有当条带内部已经出现 Cloudflare 组件笔画、logo、勾选框边缘等内容时，才进入点击阶段

这一步能避免：

- 空白占位时误点
- 把 challenge 点进错误刷新状态

## 3. 识别真实复选框时，不要只依赖 DOM

这类站点常见现象：

- DOM 中 iframe 是 `1x1`
- 真实挑战内容在外层封装或 Shadow DOM 内
- hidden input 是可见的，但实际可点区域不是

因此建议组合使用：

1. DOM / hidden input 邻域定位 challenge 条带
2. 基于截图局部做 OpenCV 方形轮廓识别
3. 用“密码框下沿 ~ 登录按钮上沿”的带状区域做垂直约束

优先过滤掉：

- 页面标题中的圆点 / 小图标
- 表单左侧装饰性方块
- 文案笔画形成的伪方形轮廓

推荐约束条件：

- 轮廓尺寸处于小方框范围
- 长宽比接近 1:1
- 位于 challenge 横条左侧
- 位于密码框下沿与登录按钮上沿之间

## 4. 没识别到真实复选框时，一律不点

这是这次实战里最重要的经验之一。

错误做法：

- 先找大容器
- 再在容器左边拍脑袋扫 3~7 个点

这种做法很容易：

- 点到标题
- 点到说明文字
- 点到错误遮罩触发区
- 把 challenge 点坏

正确做法：

- 如果 `checkbox_cv` 没锁定成功，就继续下一轮等待
- 允许“等待更久”，不要允许“盲点更多”

## 5. 真实复选框只点一次

如果已经识别到真实复选框：

- 默认只点击一次中心点
- 点击后等待一段时间观察 token 是否生成
- 不要在同一轮里继续补第 2、第 3 次点击

原因：

- Cloudflare 点击后可能已经进入“验证中”
- 后续补点很容易把 challenge 点进错误遮罩或异常刷新状态

建议点击策略：

1. 模拟真人鼠标移动到复选框中心
2. 单击一次
3. 等待 7~10 秒
4. 再看 hidden token 是否回填

## 6. 以 hidden token 回填作为权威成功信号

对这类站点，优先检查：

- `input[name="cf-turnstile-response"]`
- `textarea[name="cf-turnstile-response"]`
- `id` 类似 `cf-chl-widget-xxxx_response`

推荐权威成功信号：

- 字段存在且 value 长度大于 0

不要只用以下信号判定成功：

- 视觉上好像点中了
- checkbox 似乎有动画
- 页面没有报错

在实战里，最稳的判定是：

- `hiddenValueLen > 0`

## 7. Cloudflare 错误遮罩要当成“可恢复状态”

如果点击复选框后出现类似：

- `Cloudflare人机验证服务加载失败`
- `请检查你到 https://challenges.cloudflare.com/ 的连接是否正常`

不要直接把整条链路判死。

推荐恢复流：

1. 检测遮罩层文案
2. 定位 `OK` / `确定` 按钮
   - 优先 DOM 可见按钮
   - 若 DOM 难定位，再用几何兜底点击中间按钮区域
3. 点击关闭遮罩
4. 确认遮罩消失
5. **刷新登录页或同域页**
6. 等待新的 challenge 重新渲染
7. 再重新识别真实复选框

特别注意：

- 关闭遮罩后，不要继续沿用旧 challenge
- 旧 challenge 很可能已经失效，必须重建

## 8. challenge 失败后要刷新重建，而不是原地补点

当出现以下任意一种情况时，应优先刷新重建：

- 点击后出现错误遮罩
- token 长时间不生成，但 challenge 已明显进入异常态
- 组件文案或边缘形态和正常 challenge 不一致

建议做法：

1. 关闭错误遮罩
2. 刷新登录页
3. 等 3~5 秒
4. 再重新走：
   - 视觉就绪判断
   - 真实复选框识别
   - 单击一次
   - 等待 token

## 9. 容器 / Xvfb / 青龙环境的额外经验

对青龙这类 Linux 容器环境：

- 优先使用真实浏览器 + Xvfb 的“有头渲染到虚拟显示器”
- 避免依赖 `easyocr/torch` 这一类重量推理库；低指令集 CPU 上容易触发 `Illegal instruction`
- 若 `nodriver` 直接启动浏览器失败，可退回“手动拉起 Chrome 再接管”
- 手动拉起时建议：
  - 独立 `user-data-dir`
  - 打开 DevTools 日志文件
  - 接管失败时带上日志路径，便于二次排障

对偶发性浏览器断连：

- 若报 `websockets.exceptions.ConnectionClosedError`
- 或 `no close frame received or sent`

可把它视为浏览器级偶发故障，优先：

1. 关闭当前浏览器
2. 清理手动 profile
3. 重开浏览器
4. 重试当前账号一次

## 10. 这类站点的推荐状态机

推荐把登录阶段拆成如下状态机：

1. 打开登录页
2. 检查 hidden input / 条带容器
3. 判断 challenge 是否视觉就绪
4. 识别真实复选框
5. 若未识别到真实复选框：
   - 继续等待
6. 若识别到真实复选框：
   - 单击一次
   - 等待 token
7. 若 token 生成：
   - 填表单
   - 提交登录
8. 若弹错误遮罩：
   - 点 `OK`
   - 刷新页
   - 回到第 2 步
9. 若浏览器断连：
   - 重开浏览器
   - 重试当前账号

## 11. 适用边界

这套参考适用于：

- 页面里存在可见 Turnstile 复选框
- 成功信号会回填 hidden token
- 业务站点允许通过视觉点击完成 challenge

不适用于：

- 需要完整 puzzle / 图片拖拽 / 旋转挑战
- 完全不可见、必须走 JS API / 自渲染 token 的站点
- 登录接口只接受浏览器运行时对象里的 challenge 结果，且页面内没有稳定可点击复选框
